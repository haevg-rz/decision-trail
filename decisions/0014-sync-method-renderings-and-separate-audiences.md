# ADR-0014: Single-source the method text across renderings, and split it into three registers (guide / spec / agent guidance)

- Status: accepted
- Date: 2026-06-28
- Amends: ADR-0007 (re-frames the single source of truth: the lean spec in `working-method.md` is canonical, this repo's `AGENTS.md` becomes a derived rendering), ADR-0010 (splits the method body — the didactic guide moves to `docs/guide.md`, leaving `working-method.md` the lean spec)
- Promoted from: [idea 0006](../ideas/0006-documentation-flaw-sync-and-audience.md)
- Amended by: [ADR-0032](0032-audience-forked-sections-are-a-declared-delta-class.md)

## Context

Idea 0006 noticed two related flaws in the method's own documentation.

**1. Two method texts have drifted out of sync.** The method body exists twice:

- in this repo, `AGENTS.md` *is* the method text (ADR-0007: AGENTS.md is the
  single source of truth here); and
- in the `starter/`, `docs/working-method.md` *is* the method text (ADR-0010:
  an adopter keeps the method in `working-method.md`; their `AGENTS.md` only
  points to it).

These are two **renderings of one logical method**, but nothing declares that
relationship, says which is authoritative, or enumerates how they are *allowed*
to differ. So they have diverged in both intended and unintended ways:

- *Intended* deltas: paths (repo-root families `ideas/` here vs `docs/`-prefixed
  `docs/ideas/` there); construction-ADR cross-references (`ADR-0011`, `ADR-0012`)
  present here, absent in the adopter copy; the "Based on decision-trail vX.Y"
  provenance citation only in the adopter copy.
- *Unintended* drift: the ADR bullet in `working-method.md` mentions the classic
  template / Decision Drivers / Considered Options, but `AGENTS.md` does not;
  `overview.md` is described with different detail in each; wording in "How to
  start working" and the contract's promise #8 differ. No discipline keeps a
  change to one flowing into the other.

**2. Three *registers* are blended in one text.** The single method body is
trying to serve three readers at once, which pull in opposite directions:

- a **guide / manual** for a *newcomer human* — the idea told well, the
  vocabulary explained, the rules narrated, in a teaching order. Prose and
  motivation are essential here.
- a **reference / spec** for a *returning human and an agent* — the terse,
  greppable tables: contract, lifecycle, layout, cross-link vocabulary, rules,
  how to start. This is exactly what an agent loads.
- **agent operating guidance** for an *agent* — the confirmation guard, "keep
  `overview.md` current", "the method is settled" — instructions *to an agent*,
  not a description of the method.

The guide and the spec pull against each other: a newcomer needs narrative and a
suitable reading order; an agent (and a returning human) just needs the tables,
and the motivating prose is wasted overhead — even a real context/token cost for
an agent (Economy #2). A single file serving both is always a compromise.

The agent register is also *misplaced and incomplete*: in this repo the agent
guidance sits in a "Working in this repo" section at the end of `AGENTS.md`, but
in the `starter/` it is **absent entirely** — an adopting project's agent
receives no confirmation guard and no keep-the-overview-current rule. So the
blend is also a *gap*: the one part that is genuinely agent-only is the part
adopters don't get.

These concerns are one problem viewed twice: deciding *what each file is for*
(audience) is what lets us say *which content is shared, which may differ, and
how the copies stay aligned* (sync).

### Considered options

**For sync (the two renderings):**

- **A — Leave them as independent hand-maintained files.** Cheapest now; but the
  drift already observed is the steady-state, and it violates Economy (#2) every
  time the method changes (edit twice, by hand, hoping they match).
- **B — Declare one canonical rendering and treat the other as *derived*,
  regenerated wholesale (never patched alone), exactly as `overview.md` already
  is (ADR-0011).** A change to the method is made once in the canonical text and
  the other rendering is regenerated by mechanically applying the enumerated
  deltas. Costs a stated regeneration discipline; buys a single edit point and a
  closed list of legitimate differences.
- **C — Physically single-source with an include/transclusion build step.** A
  generator stitches one body into both files. Most rigorous, but it introduces
  tooling and a build artifact — directly against Borrow-don't-invent (#8) and
  Transparency (#3, plain readable markdown, nothing generated by a tool you
  must run).

**For audience (the three registers):**

- **D — Keep the blend; add clearer section headers only.** Minimal; leaves both
  the guide/spec compromise and the adopter agent-guidance gap unsolved.
- **E — Two-way split: a neutral *method description* (guide + spec together) and
  a distinct *agent operating guidance* block** shipped to adopters. Fixes the
  agent gap, but still asks one document to be both newcomer-narrative and
  agent-loaded spec.
- **F — Three-way split by register: a dedicated human *Guide*, a lean *Reference
  / spec* (what the agent pointer targets), and an *agent operating guidance*
  block** shipped to adopters. The Guide carries the prose and teaching order; the
  spec stays terse and greppable, so an agent loads only what it needs. Cost: one
  more file on the sync surface this ADR is otherwise trying to shrink.

### Decision drivers

- **Economy (#2):** the method changes during the use phase (this ADR is itself
  such a change); editing the body in one place beats editing two by hand.
- **Transparency (#3):** stay in plain, readable markdown — no build step, no
  generated-by-tool artifact. This rules out C.
- **Continuity (#1) for adopters:** an adopting repo must be self-contained,
  including the agent rules its agent needs — closing the current gap.
- **Right register for each reader:** a newcomer human needs narrative and a
  teaching order; an agent needs only the terse spec and should not load the
  motivating prose (Economy #2, both for the human's onboarding cost and the
  agent's context cost).
- **Consistency with settled mechanics:** `overview.md` already establishes the
  "one source, regenerate the derived view, never hand-patch" pattern (ADR-0011);
  reusing it for the method renderings is the same move, not a new invention.

## Proposed decision

Adopt **B + F**.

1. **One canonical method text; the other rendering is derived.** The canonical
   method body is **`starter/docs/working-method.md`** — the generic, unbranded,
   adopter-facing rendering. This repo's `AGENTS.md` method body is a **derived
   rendering** of it, produced by applying a fixed, enumerated set of deltas, and
   **regenerated wholesale, never hand-patched** (the `overview.md` discipline,
   ADR-0011, applied to prose).

   > Choosing the *adopter* copy as canonical keeps the generic text primary and
   > this repo's self-references (construction-ADR links) as the *added* layer —
   > the brand/construction footprint stays an overlay, never the base
   > (Genericity #7). This is the main point to confirm or flip.

2. **Enumerate the allowed deltas** (the only legitimate differences between the
   two renderings), recorded in the canonical file's header note:
   - artifact paths: repo-root families in this repo vs `docs/`-prefixed in an
     adopter;
   - construction-ADR cross-references (`ADR-00NN`) present in this repo's
     rendering, absent in the adopter's;
   - the `Based on decision-trail vX.Y` provenance citation, adopter-only;
   - the entry-point framing (this repo: `AGENTS.md` *is* the text; adopter:
     `AGENTS.md` points to `docs/working-method.md`).

3. **A one-line sync note in each rendering's header** stating its role
   ("canonical" / "derived — regenerate from `working-method.md`, do not patch
   alone"), so the relationship is visible on sight.

4. **Split the method text into three registers.**
   - **Guide (`docs/guide.md`)** — human-facing and didactic: the idea told well,
     the vocabulary explained in prose, the rules narrated, in a suitable teaching
     order. This is the newcomer's manual. decision-trail's *own* origin story
     (how it arose, my-second-brain) is this-repo history and stays in this
     repo's `README`/`AGENTS.md`, **not** in the generic adopter Guide — the
     adopter Guide teaches *using* the method.
   - **Reference / spec (`docs/working-method.md`)** — lean and greppable: the
     contract, lifecycle, layout, cross-link vocabulary, rules, how to start.
     This is the canonical method text of decision 1 above, and the target the
     `AGENTS.md` "How we work" pointer resolves to, so an agent loads only the
     terse spec, not the Guide's prose.
   - **Agent operating guidance** — the agent-only rules (confirmation guard,
     "keep `overview.md` current", "the method is settled"), in a distinct,
     clearly-headed block carried in `AGENTS.md`, and **included in the
     `starter/AGENTS.md`** so adopting projects' agents receive it (closing
     today's gap).

   The Guide links to the spec and the agent-guidance block; none of the three
   duplicates another's content.

## Consequences

- A change to the method is edited once in `starter/docs/working-method.md`, then
  the `AGENTS.md` rendering is regenerated by applying the enumerated deltas —
  one source of truth for the prose, a closed list of legitimate differences
  (Economy #2).
- No tooling or build step is introduced; both files remain plain, directly
  readable markdown (Transparency #3, Borrow-don't-invent #8).
- Adopting repos gain the agent operating guidance they currently lack, and a
  dedicated human Guide for onboarding, restoring self-containedness and lowering
  the newcomer's cost-to-start (Continuity #1, Economy #2).
- Each reader meets the right register: a newcomer reads the Guide; a returning
  human or an agent loads only the lean spec; the agent also gets its operating
  rules. The motivating prose no longer burdens the agent's context.
- Cost: the Guide adds one file to keep aligned. It is mostly stable narrative
  (the idea, the vocabulary, the rules) that changes rarely, and it is *not* a
  two-rendering artifact with deltas the way the spec is — so its sync burden is
  far lighter than the spec's.
- This refines, and partially re-frames, ADR-0007 (which named *this repo's*
  `AGENTS.md` the single source of truth) and builds on ADR-0010/ADR-0011. If
  accepted, the cross-links (`Amends:` to ADR-0007, and to ADR-0010/ADR-0011 as
  appropriate) are finalized at acceptance, and the carry-out work — writing the
  `docs/guide.md` Guide, slimming `working-method.md` to the lean spec, splitting
  out the agent-guidance block, writing the header sync notes, regenerating
  `AGENTS.md`, and updating `starter/AGENTS.md` — belongs to a follow-up plan that
  `Implements:` this ADR, not to this decision.
- Open question for acceptance: confirm the **canonical direction** (adopter copy
  canonical, as proposed, vs. this repo's `AGENTS.md` canonical).
